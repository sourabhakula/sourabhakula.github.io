The Problem

Raw CSVs are not analytics. How do you build a pipeline that is maintainable, tested, and documented?

Working with 100,000+ real Olist (Brazilian e-commerce) transactions across 9 CSV seed files covering orders, customers, sellers, payments, reviews, and product categories. The challenge: transform raw, messy CSVs into clean, tested, documented analytical tables that a BI tool could query reliably.

The Insight

dbt's ref() function changes how you think about SQL

Key Insight
"The shift from 'writing queries' to 'building data flows' is the difference between ad-hoc analysis and analytics engineering. dbt forces that shift by making dependencies explicit."

Instead of writing individual queries, you start thinking in data flows. Dependency tracking, automated testing, and self-generating documentation make a pipeline maintainable in a way raw SQL never is. Each model is a node in a graph, not a standalone script.

The Solution

A layered dbt pipeline: staging, intermediate, marts

  • 01
    Staging layer: Cleans and types raw CSVs
  • 02
    Intermediate layer: Joins tables and applies business logic
  • 03
    Marts layer: Three final tables ready for BI tools covering order performance, seller performance, and customer behavior
  • 04
    17 data tests covering uniqueness, nulls, and accepted values
  • 05
    dbt docs generate builds full lineage graph automatically

What I Learned

  • Layered architecture prevents spaghetti SQL. Staging/intermediate/marts separation means each layer has one job. Changes propagate cleanly.
  • Testing is not optional in data engineering. 17 tests caught issues that would have silently corrupted downstream dashboards. Tests are cheap insurance.
  • Documentation that generates itself actually gets maintained. dbt's auto-generated lineage docs solved the age-old problem of documentation drifting from reality.
